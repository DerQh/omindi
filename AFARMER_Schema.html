2. Database Schema

All tables live in the public schema of the Supabase PostgreSQL database. Primary keys are PK, foreign keys are FK.

profiles

One row per authenticated user. Created automatically on sign-up via trigger. Links to auth.users.

Column Type Notes
id uuid PK — matches auth.users.id
full_name text
farm_name text
avatar_url text URL from avatars storage bucket
bio text
location text
phone text
is_admin boolean Default false. Grants admin panel access
created_at timestamptz Default now()

listings

Marketplace produce/product listings created by sellers.

Column Type Notes
id uuid PK
seller_id uuid FK → profiles.id
seller_name text Denormalized for display speed
seller_image_url text Denormalized avatar
title text
description text
price numeric
unit text e.g. "kg", "litre"
category text
location text
minimumOrder text
image_url text From listing-images bucket
phone text Contact number for listing
available boolean Stock status. Default true
views integer Incremented via RPC
inquiries integer Incremented via RPC
favourites integer Incremented/decremented via RPC
created_at timestamptz Default now()

orders

A buyer's purchase order. One order can have many order_items.

Column Type Notes
id uuid / bigint PK
user_id uuid FK → profiles.id (buyer)
total_cost numeric
payment_method text e.g. "mpesa", "cash"
delivery_address text
mobile_no text M-Pesa number
status text "pending" | "processing" | "confirmed" | "delivered" | "cancelled" | "payment_failed"
admin_note text Admin dispute notes
mpesa_checkout_id text Safaricom STK push CheckoutRequestID — used to match callback
mpesa_receipt text M-Pesa transaction receipt code (set when payment succeeds)
mpesa_result_desc text Safaricom result description (populated on failure for display)
paid_at timestamptz Timestamp when payment was confirmed (M-Pesa or Stripe)
created_at timestamptz Default now()

Status flow: pending → confirmed (payment success) or payment_failed (payment failed) → processing → delivered / cancelled.
Set by: M-Pesa status is updated by the mpesa-callback Edge Function. Stripe status is updated inside Checkout.jsx after stripe.confirmCardPayment succeeds.

order_items

Line items within an order. Stores price at purchase time to prevent drift.

Column Type Notes
id bigint PK
order_id uuid / bigint FK → orders.id
listing_id uuid FK → listings.id
quantity integer
price_at_purchase numeric Snapshot of price when ordered

cart

Per-user shopping cart. Cleared after checkout.

Column Type Notes
id bigint PK
user_id uuid FK → profiles.id
listing_id uuid FK → listings.id
quantity integer Default 1
created_at timestamptz

reviews

Buyer reviews for listings and shop items. One review per user per target enforced by unique partial indexes.

Column Type Notes
id bigint PK GENERATED ALWAYS AS IDENTITY
listing_id uuid FK → listings.id ON DELETE CASCADE (nullable)
shop_item_id text Static shop product ID (nullable)
seller_id uuid FK → profiles.id ON DELETE SET NULL
reviewer_id uuid FK → profiles.id ON DELETE CASCADE — NOT NULL
rating smallint CHECK (rating BETWEEN 1 AND 5)
comment text Optional
created_at timestamptz Default now()

transactions

Payment records (M-Pesa or other). Linked to orders.

Column Type Notes
id bigint PK
user_id uuid FK → profiles.id
order_id uuid / bigint FK → orders.id
payment_method text
amount numeric
phone text M-Pesa phone number
status text "pending" | "approved" | "failed"
reference text Payment reference/code
updated_at timestamptz

listing_favorites

Junction table — which users have bookmarked which listings.

Column Type Notes
id bigint PK
user_id uuid FK → profiles.id
listing_id uuid FK → listings.id

follows

User follow relationships (follower → following).

Column Type Notes
id bigint PK
follower_id uuid FK → profiles.id
following_id uuid FK → profiles.id
created_at timestamptz

conversations

Buyer ↔ seller messaging threads, scoped to a listing.

Column Type Notes
id uuid / bigint PK
buyer_id uuid FK → profiles.id
seller_id uuid FK → profiles.id
listing_id uuid FK → listings.id
created_at timestamptz

messages

Individual chat messages inside a conversation.

Column Type Notes
id bigint PK
conversation_id uuid / bigint FK → conversations.id
sender_id uuid FK → profiles.id
content text
read boolean Default false
created_at timestamptz

notifications

In-app notifications. Types: order, favorite, follow, message, broadcast.

Column Type Notes
id bigint PK
user_id uuid FK → profiles.id (recipient)
type text "order" | "favorite" | "follow" | "message" | "broadcast"
title text
body text

posts

Community feed posts (text + optional image).

Column Type Notes
id bigint PK
user_id uuid FK → profiles.id
content text

created_at timestamptz

post_likes

Junction — which users liked which posts.

Column Type Notes
id bigint PK
post_id bigint FK → posts.id
user_id uuid FK → profiles.id

post_comments

Comments on community posts.

Column Type Notes
id bigint PK
post_id bigint FK → posts.id

Admin-managed merchandise (DB-backed items, separate from static shopProducts.js).

Column Type Notes

Peer-to-peer user ratings (distinct from listing reviews).

Column Type Notes

rating numeric 1–5
UNIQUE — (rater_id, rated_user_id)

4. Supabase Edge Functions

All functions live in supabase/functions/. Deploy with supabase functions deploy <name>. Secrets are set in Supabase Dashboard → Edge Functions → Secrets (never in .env).

Function Trigger Purpose Key Secrets Used
mpesa-stk-push Called from useMpesaPay hook on checkout Obtains Daraja OAuth token, builds STK push payload, sends payment prompt to customer's phone. Saves mpesa_checkout_id to order row. MPESA_CONSUMER_KEY, MPESA_CONSUMER_SECRET, MPESA_SHORTCODE, MPESA_PASSKEY, MPESA_CALLBACK_URL
mpesa-callback Safaricom Daraja HTTPS callback (public endpoint) Receives payment result from Safaricom. On success: sets order status to confirmed, saves mpesa_receipt and paid_at. On failure: sets status to payment_failed, saves mpesa_result_desc. SUPABASE_SERVICE_ROLE_KEY
mpesa-stk-query Called from usePollOrderStatus as a fallback after callback timeout Queries Safaricom Daraja for the status of a specific STK push request (by CheckoutRequestID). Used when the callback hasn't arrived after a few seconds. MPESA_CONSUMER_KEY, MPESA_CONSUMER_SECRET, MPESA_SHORTCODE, MPESA_PASSKEY
create-payment-intent Called from Checkout.jsx when payment method = "card" Creates a Stripe PaymentIntent for the order amount (in KES × 100 = smallest currency unit). Returns clientSecret to the browser, which then calls stripe.confirmCardPayment — card data never touches our servers. STRIPE_SECRET_KEY
send-order-email Called from Checkout.jsx on successful payment (M-Pesa or Stripe) Fetches order + items from DB, renders an HTML email, sends via Resend API to the buyer's email. Reply-to is difffred@gmail.com. RESEND_API_KEY, SUPABASE_SERVICE_ROLE_KEY

9. Auth Flow

How a user goes from first visit to accessing protected app features.

Step What happens Code location
1. Visit /sign-up or /login User enters email + password. Supabase Auth handles sign-up or sign-in. SignupLogin.jsx
2. Profile auto-created A Supabase trigger fires on auth.users INSERT and creates a matching row in public.profiles DB trigger: handle_new_user()
3. Session stored Supabase stores a JWT in localStorage. All subsequent requests are authenticated via this token. supabase.js (createClient)
4. useUser() hook Calls supabase.auth.getUser() + fetches profiles row. Returns the full user object to any component. hooks/useUser.js
5. Protected routes ProtectedRoutes.jsx wraps all /app/* routes. If no session → redirect to /login. components/mobile/ProtectedRoutes.jsx
6. Admin check profiles.is_admin = true grants access to /admin. Checked inside Admin.jsx on mount. hooks/useAdmin.js, Admin.jsx
7. Password reset User requests reset → Supabase sends email → user clicks link → /reset-password page handles the token. ResetPassword.jsx
8. Sign out supabase.auth.signOut() clears session. React Query cache is not automatically cleared — page refresh recommended. AppNavbar.jsx / Navbar.jsx

10. Data Flow — Key Journeys

Cart → Order → Payment (M-Pesa STK Push)

Step Action Tables touched Where
1 Buyer taps "Add to Cart" cart INSERT useAddItem
2 Buyer goes to /checkout, selects M-Pesa, fills phone + delivery address — Checkout.jsx
3 Order placed (status: pending) orders INSERT useCreateOrder
4 Line items saved order_items INSERT (one per cart item) useCreateOrderItem
5 M-Pesa STK Push triggered — push prompt sent to buyer's phone orders UPDATE (mpesa_checkout_id) useMpesaPay → mpesa-stk-push Edge Function
6 Buyer approves on phone → Safaricom calls callback URL orders UPDATE (status: confirmed, mpesa_receipt, paid_at) mpesa-callback Edge Function
6b If callback delayed → STK query fallback checks status orders UPDATE (same as step 6) mpesa-stk-query Edge Function
7 Order confirmation email sent — send-order-email Edge Function (via Resend)
8 Cart cleared cart DELETE (all rows for user) useCart / Checkout.jsx
9 Seller notified notifications INSERT (type: order) useOrders onSuccess

Cart → Order → Payment (Stripe Card)

Step Action Tables touched Where
1–4 Same as M-Pesa steps 1–4 (cart → order → items) cart, orders, order_items Same hooks
5 Buyer enters card details via Stripe CardElement + name on card — Checkout.jsx (Stripe Elements — card data never hits our servers)
6 PaymentIntent created server-side — create-payment-intent Edge Function → returns clientSecret
7 Card payment confirmed by Stripe orders UPDATE (status: confirmed, paid_at: now()) Checkout.jsx → stripe.confirmCardPayment
8–9 Email + cart clear + seller notification Same as M-Pesa steps 7–9 Same

Listing → Review

Step Action Tables touched Hook
1 Seller creates listing listings INSERT + listing-images storage upload useCreateListing
2 Buyer views listing listings UPDATE (views + 1 via RPC) ListingDetail useEffect
3 Buyer bookmarks listing listing_favorites INSERT + listings UPDATE (favourites + 1 via RPC) useCreateFavorite
4 Buyer sends inquiry conversations INSERT + listings UPDATE (inquiries + 1 via RPC) useStartConversation
5 Buyer submits review (after delivery) reviews INSERT useAddReview
6 Seller rating updates reviews SELECT (avg computed client-side) useSellerRating

Post → Community Interaction

Step Action Tables touched Hook
1 User creates post (text + optional image) posts INSERT + post-images storage useCreatePost / useLists
2 Another user likes post post_likes INSERT usePostLike
3 Another user comments post_comments INSERT usePostComment
4 User follows another user follows INSERT + notifications INSERT (type: follow) useFollows

18. Business Rules

The platform laws — enforced in code and/or RLS. These must never be accidentally broken.

RuleWhere enforced
Only the seller who created a listing can edit or delete itRLS policy on listings + UI check (isSeller)
A user cannot review their own listingUI: !isSeller hides review form
One review per user per listing (and per shop item)Unique partial index on reviews + useHasReviewed check
Cart is per-user — you cannot see another user's cartRLS: auth.uid() = user_id
Messages only visible to buyer and seller in that conversationRLS: buyer_id = auth.uid() OR seller_id = auth.uid()
A user can only follow each person onceUNIQUE constraint on (follower_id, following_id)
A user can only bookmark each listing onceUNIQUE constraint on (user_id, listing_id)
Favourites counter never goes below 0RPC: GREATEST(favourites - 1, 0)
Ratings must be between 1 and 5CHECK constraint on reviews.rating and user_ratings.rating
Admin can delete any listing, order, or reviewRLS admin policy: is_admin = true in profiles
Price at purchase is frozen in order_itemsprice_at_purchase column — copied at order creation, never linked back to listings.price
Protected routes require an active Supabase sessionProtectedRoutes.jsx redirects to /login if no session
Peer ratings upserted — one rating per user pairUNIQUE (rater_id, rated_user_id) + upsert with onConflict

-- Auto-create profile on sign-up
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS trigger LANGUAGE plpgsql SECURITY DEFINER AS $$
BEGIN
  INSERT INTO public.profiles (id, full_name)
  VALUES (NEW.id, NEW.raw_user_meta_data->>'full_name')
  ON CONFLICT (id) DO NOTHING;
  RETURN NEW;
END;
$$;
CREATE OR REPLACE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();

-- Auto-create profile on sign-up (trigger)
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS trigger LANGUAGE plpgsql SECURITY DEFINER AS $$
BEGIN
  INSERT INTO public.profiles (id, full_name)
  VALUES (NEW.id, COALESCE(NEW.raw_user_meta_data->>'full_name', NEW.raw_user_meta_data->>'username', ''))
  ON CONFLICT (id) DO NOTHING;
  RETURN NEW;
END;
$$;
DROP TRIGGER IF EXISTS on_auth_user_created ON auth.users;
CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();
-- Full-text search on listings
ALTER TABLE public.listings
  ADD COLUMN IF NOT EXISTS search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(description, '')), 'B') ||
    setweight(to_tsvector('english', coalesce(category, '')), 'C') ||
    setweight(to_tsvector('english', coalesce(location, '')), 'C')
  ) STORED;

CREATE INDEX IF NOT EXISTS listings_search_idx ON public.listings USING GIN(search_vector);

CREATE OR REPLACE FUNCTION public.search_listings(search_query text)
RETURNS SETOF public.listings LANGUAGE sql STABLE AS $$
  SELECT * FROM public.listings
  WHERE search_vector @@ plainto_tsquery('english', search_query)
  ORDER BY ts_rank(search_vector, plainto_tsquery('english', search_query)) DESC;
$$;
